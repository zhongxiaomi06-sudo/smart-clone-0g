# 需求对齐说明

这份说明用于确认当前代码框架是否按产品文档中的方向搭建。结论是：当前实现遵循了“应用级底座”的核心方向，但它还是底层框架，不是完整产品实现。

## 总体结论

当前代码不是围绕“故事生成”写死的应用，而是围绕以下底座能力搭建：

- 记忆仓库
- Chat 主入口
- Skill 注册与调用
- MCP 工具网关
- 权限与审计
- 本地优先存储
- 后续智能硬件与 Web3 的接口预留

故事生成只是 `skills/daily_story` 中的一个示例 Skill，可以删除、替换或新增同类 Skill，不影响核心框架。

## 需求映射

| 文档要求 | 当前实现 | 状态 |
| --- | --- | --- |

| 记忆仓库作为底座 | `src/smart_avatar/storage.py` 提供记忆卡片、状态卡片、权限、审计的本地 SQLite 存储 | 已落实 |
| 故事生成不要作为主入口 | 故事生成只存在于 `skills/daily_story` 配置目录，核心代码不写死故事逻辑 | 已落实 |
| Skill 作为接口能力 | `src/smart_avatar/skills.py` 根据 `skill.json` 加载 Skill，并按权限运行 | 已落实 |
| 支持外部场景应用 Skill | 新 Skill 只需新增 `skills/{name}/skill.json` 和 `prompt.md` | 已落实 |
| 多加接口，功能向外拓展 | 已提供记忆查询、状态卡片、Chat、Skill、工具、权限、审计 API | 已落实 |

## 当前不是完整实现的部分

以下能力还没有真正实现，不能在产品说明里当成已完成：

- 本地语音转写。
- 本地小模型脱敏提炼。
- 向量数据库和语义检索。
- 真实大模型调用。
- 真实 MCP Server 调用。
- 米家、手表、手环等硬件数据接入。
- 真实链上提交、DID、零知识证明。
- 前端界面。
- 用户登录、多设备同步和密钥恢复。

当前代码完成的是应用底座和接口边界，让这些能力可以按模块接入。

## 架构约束

后续开发必须遵守以下约束：

1. 不把故事生成、周复盘、身心分析等垂直场景写进核心 Chat 或 Storage 层。
2. 新场景必须优先作为 Skill 接入。
3. 外部工具必须通过 MCP 网关或工具适配器接入，不能直接读取完整记忆库。
4. Skill 读取记忆必须声明权限、用途和数据范围。
5. 原始语音、原始文本、身体明细和故事全文不能上链。
6. 链上只允许保存哈希、授权、审计或零知识证明结果。
7. Chat 中枢是用户主入口，Skill 是被调用能力。
8. 模型提供商、存储后端、检索方式和工具协议都必须可替换。

## 当前最小可用闭环

当前框架支持的最小闭环是：

1. 写入脱敏记忆卡片。
2. 通过 Chat 查询记忆。
3. Chat 根据触发词匹配 Skill。
4. Skill 请求权限确认。
5. 用户确认后 Skill 读取授权范围内的记忆。
6. 模型层生成结果，目前使用 dry-run 占位。
7. 系统写入审计日志。

这个闭环符合“先垂直搭基础，再向外扩展”的方向。
